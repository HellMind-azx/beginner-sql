Основы и DDL

Data Definition Language · CREATE TABLE · Типы данных · Ограничения

Теория Что такое SQL и DDL? ▼

SQL — язык для общения с базой данных. Он делится на 3 части:

DDL
Data Definition Language
CREATE, ALTER, DROP — создаём структуру
DML
Data Manipulation Language
INSERT, UPDATE, DELETE, SELECT — работаем с данными
DCL
Data Control Language
GRANT, REVOKE — права доступа

Сегодня — только DDL. Это аналог models.py в Django. Ты описываешь структуру таблиц вручную, без ORM.

🐍 Django аналогия
Django class Post(models.Model) = SQL CREATE TABLE post (...)
Django CharField(max_length=200) = SQL VARCHAR(200)
Django null=False = SQL NOT NULL
Теория Ключевые типы данных PostgreSQL ▼
Текст
VARCHAR(n) — строка до n символов
TEXT — строка любой длины
CHAR(n) — фиксированная длина
Числа
INTEGER — целое число
BIGINT — большое целое
NUMERIC(p,s) — точное дробное
Дата / Время
DATE — только дата
TIMESTAMP — дата + время
TIMESTAMPTZ — + часовой пояс ✓
Прочее
BOOLEAN — true/false
SERIAL — авто-инкремент
UUID — уникальный ID
🐍 Django аналогия
models.CharField → VARCHAR(n)
models.TextField → TEXT
models.IntegerField → INTEGER
models.DateTimeField(auto_now_add=True) → TIMESTAMPTZ DEFAULT NOW()
Теория Ограничения (Constraints) ▼
PRIMARY KEY — уникальный идентификатор строки. Каждая таблица должна иметь один. Django: id = models.AutoField(primary_key=True)
NOT NULL — поле не может быть пустым. Django: null=False (по умолчанию)
UNIQUE — значение уникально в колонке. Django: unique=True
DEFAULT — значение по умолчанию. Django: default=...
CHECK — произвольное условие. Django: validators=[...]
Практика SQL-команды для отработки ▼

Запусти PostgreSQL (psql или DBeaver) и выполни по порядку:

-- 1. Создаём базу данных для блога
CREATE DATABASE blog_db;
\c blog_db  -- подключаемся (в psql)

-- 2. Создаём таблицу пользователей
CREATE TABLE users (
    id        SERIAL        PRIMARY KEY,
    username  VARCHAR(50)   NOT NULL UNIQUE,
    email     VARCHAR(255)  NOT NULL UNIQUE,
    bio       TEXT,
    is_active BOOLEAN       DEFAULT TRUE,
    joined_at TIMESTAMPTZ   DEFAULT NOW()
);

-- 3. Смотрим структуру таблицы
\d users
-- 4. Создаём таблицу статей (посмотрим на CHECK)
CREATE TABLE posts (
    id           SERIAL       PRIMARY KEY,
    title        VARCHAR(200) NOT NULL,
    slug         VARCHAR(200) NOT NULL UNIQUE,
    content      TEXT         NOT NULL,
    status       VARCHAR(20)  NOT NULL DEFAULT 'draft'
                 CHECK (status IN ('draft', 'published', 'archived')),
    views_count  INTEGER      DEFAULT 0,
    published_at TIMESTAMPTZ,
    created_at   TIMESTAMPTZ  DEFAULT NOW()
);

-- 5. Создаём таблицу тегов
CREATE TABLE tags (
    id   SERIAL      PRIMARY KEY,
    name VARCHAR(50) NOT NULL UNIQUE
);
-- 6. Если допустил ошибку — можно удалить и пересоздать
DROP TABLE IF EXISTS posts;

-- 7. Добавить колонку к существующей таблице
ALTER TABLE users ADD COLUMN avatar_url TEXT;

-- 8. Список всех таблиц в psql
\dt
🐍 Что это значит в Django
Когда ты запускаешь python manage.py migrate, Django под капотом выполняет именно эти CREATE TABLE команды. Сейчас ты делаешь то, что Django скрывал от тебя!

02

DML — Работа с данными

INSERT · UPDATE · DELETE · SELECT · WHERE

Теория INSERT — добавляем данные ▼
🐍 Django аналогия
User.objects.create(username="john") = INSERT INTO users (username) VALUES ('john') 
-- Одна запись
INSERT INTO users (username, email, bio)
VALUES ('john_doe', 'john@example.com', 'Backend разработчик');

-- Несколько записей сразу
INSERT INTO users (username, email) VALUES
    ('alice',   'alice@example.com'),
    ('bob',     'bob@example.com'),
    ('charlie', 'charlie@example.com');

-- Вставить и сразу вернуть созданный ID
INSERT INTO posts (title, slug, content)
VALUES ('Мой первый пост', 'my-first-post', 'Привет мир!')
RETURNING id, created_at; -- PostgreSQL-фича!
Теория SELECT и WHERE — выборка и фильтрация ▼
🐍 Django аналогия
Post.objects.all() → SELECT * FROM posts
Post.objects.get(id=1) → SELECT * FROM posts WHERE id = 1
Post.objects.filter(status='published') → SELECT * FROM posts WHERE status = 'published'
Post.objects.filter(title__contains='SQL') → SELECT * FROM posts WHERE title LIKE '%SQL%' 
-- Все записи
SELECT * FROM users;

-- Конкретные колонки
SELECT id, username, email FROM users;

-- Фильтры
SELECT * FROM posts WHERE status = 'published';
SELECT * FROM posts WHERE views_count > 100;
SELECT * FROM posts WHERE title LIKE '%Python%'; -- % = любые символы
SELECT * FROM posts WHERE status IN ('draft', 'archived');
SELECT * FROM posts WHERE views_count BETWEEN 50 AND 500;
SELECT * FROM users  WHERE bio IS NULL;    -- NULL проверка!
SELECT * FROM users  WHERE bio IS NOT NULL;

-- Сортировка и лимит
SELECT * FROM posts ORDER BY created_at DESC LIMIT 10;
Практика UPDATE и DELETE ▼
🐍 Django аналогия
Post.objects.filter(id=1).update(status='published') → UPDATE posts SET status='published' WHERE id=1
Post.objects.filter(status='draft').delete() → DELETE FROM posts WHERE status='draft' 
-- UPDATE — всегда используй WHERE!
UPDATE posts
SET    status = 'published', published_at = NOW()
WHERE  id = 1;

-- Обновить несколько полей
UPDATE users
SET    bio = 'Обновлённое описание', is_active = TRUE
WHERE  username = 'john_doe'
RETURNING *; -- вернуть обновлённую строку

-- DELETE — ОБЯЗАТЕЛЬНО с WHERE, иначе удалишь всё!
DELETE FROM posts WHERE status = 'archived';
DELETE FROM users WHERE id = 5;

Связи таблиц

FOREIGN KEY · Один-к-одному · Один-ко-многим

Теория Внешние ключи — FOREIGN KEY ▼

FOREIGN KEY — это ссылка одной таблицы на строку в другой. База сама следит за целостностью: нельзя создать пост с author_id = 999, если пользователя с id=999 не существует.

🐍 Django аналогия
author = models.ForeignKey(User, on_delete=models.CASCADE)
→ В SQL это колонка + FOREIGN KEY + ON DELETE CASCADE

profile = models.OneToOneField(User, ...)
→ FOREIGN KEY + UNIQUE (уникальный = один-к-одному) 
-- ОДИН-КО-МНОГИМ: один user → много posts
ALTER TABLE posts
ADD COLUMN author_id INTEGER REFERENCES users(id)
    ON DELETE CASCADE;
    -- CASCADE = удалить пользователя → удалятся его посты
    -- SET NULL  = удалить пользователя → author_id станет NULL
    -- RESTRICT  = запретить удаление, если есть посты

-- ОДИН-К-ОДНОМУ: один user → один profile
ALTER TABLE profiles
ADD COLUMN user_id INTEGER UNIQUE
    REFERENCES users(id) ON DELETE CASCADE;
    -- UNIQUE делает связь один-к-одному!

-- Посмотреть все внешние ключи таблицы
SELECT conname, contype, pg_get_constraintdef(oid)
FROM   pg_constraint
WHERE  conrelid = 'posts'::regclass;

JOIN — объединение таблиц

INNER JOIN · LEFT JOIN · RIGHT JOIN

Теория INNER JOIN vs LEFT JOIN ▼
INNER JOIN
Только строки, у которых есть совпадение в обеих таблицах.
Посты БЕЗ автора не попадут в результат.
LEFT JOIN
Все строки из левой таблицы + совпадения из правой.
Посты БЕЗ автора попадут, author = NULL.
🐍 Django аналогия
Post.objects.select_related('author') → SELECT * FROM posts INNER JOIN users ON posts.author_id = users.id

Post.objects.filter(author__isnull=False) ≈ INNER JOIN (только с автором) 
-- INNER JOIN: посты только с авторами
SELECT
    p.id,
    p.title,
    p.status,
    u.username  AS author_name,
    u.email     AS author_email
FROM       posts p
INNER JOIN users u ON p.author_id = u.id;

-- LEFT JOIN: все посты, даже без автора
SELECT
    p.title,
    u.username AS author_name  -- NULL если автора нет
FROM      posts p
LEFT JOIN users u ON p.author_id = u.id;

-- Тройной JOIN: посты + авторы + комментарии
SELECT
    p.title,
    u.username AS author,
    COUNT(c.id) AS comment_count
FROM      posts p
LEFT JOIN users    u ON p.author_id = u.id
LEFT JOIN comments c ON c.post_id  = p.id
GROUP BY  p.id, p.title, u.username;

Агрегация и группировка

COUNT · SUM · AVG · GROUP BY · HAVING

Теория Агрегатные функции ▼
🐍 Django аналогия
Post.objects.count() → SELECT COUNT(*) FROM posts
Post.objects.aggregate(Avg('views_count')) → SELECT AVG(views_count) FROM posts
Post.objects.values('author_id').annotate(c=Count('id')) → SELECT author_id, COUNT(*) FROM posts GROUP BY author_id 
-- Базовые агрегаты
SELECT
    COUNT(*),              -- кол-во строк
    SUM(views_count),      -- сумма просмотров
    AVG(views_count),      -- среднее
    MAX(views_count),      -- максимум
    MIN(views_count)       -- минимум
FROM posts;

-- GROUP BY: кол-во постов по каждому автору
SELECT
    u.username,
    COUNT(p.id)    AS post_count,
    SUM(p.views_count) AS total_views
FROM       users u
LEFT JOIN  posts p ON p.author_id = u.id
GROUP BY   u.id, u.username
ORDER BY   post_count DESC;

-- HAVING: фильтрация ПОСЛЕ группировки (WHERE нельзя!)
SELECT
    u.username,
    COUNT(p.id) AS post_count
FROM       users u
LEFT JOIN  posts p ON p.author_id = u.id
GROUP BY   u.id, u.username
HAVING     COUNT(p.id) > 5  -- только авторы с >5 постами
ORDER BY   post_count DESC;
⚡ Ключевое правило
WHERE фильтрует строки ДО группировки.
HAVING фильтрует группы ПОСЛЕ группировки.
Нельзя писать WHERE COUNT(*) > 5 — только HAVING COUNT(*) > 5
